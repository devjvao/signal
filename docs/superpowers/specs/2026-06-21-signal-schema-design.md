# Signal — Database Schema Design

**Date:** 2026-06-21
**Scope:** PostgreSQL schema and golang-migrate migrations for the core entities: `users`, `projects`, `feature_requests`, `votes`.

---

## Overview

This spec defines the relational schema backing Signal's authentication and feature-voting system. It covers table definitions, relationships, constraints, and indexes only — no application code (handlers, sqlc queries, auth logic) is in scope here.

Model: a user owns zero or more projects (single owner, no membership table). Each project has feature requests, created by any user. Users cast simple upvotes on feature requests.

---

## Cross-Cutting Conventions

Applied uniformly to all four tables:

- **Primary key:** `id UUID PRIMARY KEY DEFAULT gen_random_uuid()`. `gen_random_uuid()` is built into Postgres core since v13 (the project runs postgres:16-alpine), so no extension is required.
- **Timestamps:** `created_at` and `updated_at`, both `TIMESTAMPTZ NOT NULL DEFAULT now()`. `updated_at` is auto-maintained by a shared `BEFORE UPDATE` trigger (`set_updated_at()`) created in the first migration, so application code never has to remember to bump it.
- **Soft delete:** `deleted_at TIMESTAMPTZ NULL`. `NULL` means the row is active. Rows are never hard-deleted by the application; `deleted_at` is set instead.
- **Soft-delete-aware uniqueness:** any uniqueness constraint that must tolerate "delete and recreate" (user email, project slug, a user's vote on a feature request) is implemented as a **partial unique index** with `WHERE deleted_at IS NULL`, so a deleted row's value can be reused by a new row.
- **Foreign keys:** `ON DELETE CASCADE` down the ownership chain (`users` → `projects` → `feature_requests` → `votes`), covering the rare case of a hard delete.

## 000001 — `set_updated_at` trigger function

Shared utility used by every table's `BEFORE UPDATE` trigger.

```sql
CREATE FUNCTION set_updated_at() RETURNS TRIGGER AS $$
BEGIN
    NEW.updated_at = now();
    RETURN NEW;
END;
$$ LANGUAGE plpgsql;
```

Down: `DROP FUNCTION set_updated_at();`

## 000002 — `users`

```sql
CREATE TABLE users (
    id            UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    email         TEXT NOT NULL,
    password_hash TEXT NOT NULL,
    name          TEXT NOT NULL,
    created_at    TIMESTAMPTZ NOT NULL DEFAULT now(),
    updated_at    TIMESTAMPTZ NOT NULL DEFAULT now(),
    deleted_at    TIMESTAMPTZ
);

CREATE UNIQUE INDEX users_email_active_idx ON users (email) WHERE deleted_at IS NULL;

CREATE TRIGGER users_set_updated_at BEFORE UPDATE ON users
    FOR EACH ROW EXECUTE FUNCTION set_updated_at();
```

- `email`: login identifier. Unique among active (non-deleted) users only.
- `password_hash`: bcrypt/argon2 hash, never plaintext. Hashing is the application's responsibility — this column just stores the result.
- `name`: display name.

## 000003 — `projects`

```sql
CREATE TABLE projects (
    id          UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    owner_id    UUID NOT NULL REFERENCES users(id) ON DELETE CASCADE,
    name        TEXT NOT NULL,
    slug        TEXT NOT NULL,
    description TEXT,
    created_at  TIMESTAMPTZ NOT NULL DEFAULT now(),
    updated_at  TIMESTAMPTZ NOT NULL DEFAULT now(),
    deleted_at  TIMESTAMPTZ
);

CREATE INDEX projects_owner_id_idx ON projects (owner_id);
CREATE UNIQUE INDEX projects_slug_active_idx ON projects (slug) WHERE deleted_at IS NULL;

CREATE TRIGGER projects_set_updated_at BEFORE UPDATE ON projects
    FOR EACH ROW EXECUTE FUNCTION set_updated_at();
```

- `owner_id`: the single owning user. Indexed for "list my projects" queries.
- `slug`: URL-friendly identifier, generated by the application from `name`. Unique among active projects only.
- `description`: optional, nullable.

## 000004 — `feature_requests`

```sql
CREATE TABLE feature_requests (
    id          UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    project_id  UUID NOT NULL REFERENCES projects(id) ON DELETE CASCADE,
    created_by  UUID NOT NULL REFERENCES users(id) ON DELETE CASCADE,
    title       TEXT NOT NULL,
    description TEXT,
    status      TEXT NOT NULL DEFAULT 'open'
                CHECK (status IN ('open', 'planned', 'in_progress', 'completed', 'rejected')),
    created_at  TIMESTAMPTZ NOT NULL DEFAULT now(),
    updated_at  TIMESTAMPTZ NOT NULL DEFAULT now(),
    deleted_at  TIMESTAMPTZ
);

CREATE INDEX feature_requests_project_status_idx
    ON feature_requests (project_id, status) WHERE deleted_at IS NULL;
CREATE INDEX feature_requests_created_by_idx ON feature_requests (created_by);

CREATE TRIGGER feature_requests_set_updated_at BEFORE UPDATE ON feature_requests
    FOR EACH ROW EXECUTE FUNCTION set_updated_at();
```

- `status`: plain `TEXT` constrained by `CHECK`, not a native Postgres `ENUM`. Extending the allowed values later is a simple `ALTER TABLE ... DROP/ADD CONSTRAINT` migration, and sqlc maps the column to a plain Go `string` with no generated enum type to thread through the codebase.
- `(project_id, status)` composite index: the primary board query is "this project's requests filtered by status."
- `created_by`: indexed for "requests created by this user."

## 000005 — `votes`

```sql
CREATE TABLE votes (
    id                 UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    feature_request_id UUID NOT NULL REFERENCES feature_requests(id) ON DELETE CASCADE,
    user_id            UUID NOT NULL REFERENCES users(id) ON DELETE CASCADE,
    created_at         TIMESTAMPTZ NOT NULL DEFAULT now(),
    updated_at         TIMESTAMPTZ NOT NULL DEFAULT now(),
    deleted_at         TIMESTAMPTZ
);

CREATE UNIQUE INDEX votes_feature_request_user_active_idx
    ON votes (feature_request_id, user_id) WHERE deleted_at IS NULL;
CREATE INDEX votes_user_id_idx ON votes (user_id);

CREATE TRIGGER votes_set_updated_at BEFORE UPDATE ON votes
    FOR EACH ROW EXECUTE FUNCTION set_updated_at();
```

- A vote is a simple upvote: one row = one upvote, no value/direction column. Vote count for a feature request is `COUNT(*)`.
- `votes_feature_request_user_active_idx` enforces one active vote per user per feature request, and doubles as the index used for vote-count tallies (it leads with `feature_request_id`).
- `votes_user_id_idx` covers "has this user voted / list this user's votes."
- Un-voting is a soft delete (`deleted_at` set); re-voting after that inserts a new row, which the partial index allows since the prior row is no longer active.
